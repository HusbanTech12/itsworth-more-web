## 🔒 Security — Critical Audit Fixes

### Issue 1: Middleware Not Running

| Severity | 🔴 CRITICAL |
|----------|-------------|
| **Problem** | File named `proxy.ts` instead of `middleware.ts` — Next.js never loaded it. All protected routes (`/dashboard/*`, `/admin/*`, `/sell/box/checkout`, `/account/*`) had no edge-level auth protection |
| **Fix** | Renamed `proxy.ts` → `middleware.ts` |
| **Files** | `proxy.ts` → `middleware.ts` |

### Issue 2: Admin Authorization Broken

| Severity | 🔴 CRITICAL |
|----------|-------------|
| **Problem** | `requireAdmin()` in all 23 admin routes only checked `userId` exists, not admin role. **Any logged-in user could access all admin CRUD endpoints** |
| **Fix** | 1. Added `role` column (`varchar(20) default "user"`) to `users` table in `db/schema.ts` 2. Created shared `requireAdmin()` in `lib/auth.ts` that queries DB for `role === "admin"` 3. Updated all 23 admin route files to import the shared function (removed inline versions) 4. Updated Clerk webhook to set `role: "admin"` for `husbantech08@gmail.com` on user created/updated |
| **Files** | `db/schema.ts`, `lib/auth.ts` (new), 23 `app/api/admin/*/route.ts` files, `app/api/webhooks/clerk/route.ts` |

### Issue 3: NaN Order ID Bugs

| Severity | 🔴 CRITICAL |
|----------|-------------|
| **Problem** | `reinspect`, `accept-revision`, `decline-revision` used `orderId` (which is `NaN` when offer-number param like `IWM-ABC123` is passed) instead of `order.id` for DB insert/update. Operations silently failed on offer-number lookups |
| **Fix** | Changed `orderId` → `order.id` in the insert/update values/where clauses |
| **Files** | `app/api/orders/[id]/reinspect/route.ts`, `app/api/orders/[id]/accept-revision/route.ts`, `app/api/orders/[id]/decline-revision/route.ts` |

### Issue 4: Hardcoded Admin Email

| Severity | 🟠 HIGH |
|----------|---------|
| **Problem** | `"husbantech08@gmail.com"` hardcoded in client-side bundle (`Header.tsx`, `MobileNav.tsx`) to show/hide Admin nav link |
| **Fix** | Moved admin check server-side via `role` column. Client-side check remains as UI convenience only (nav link visibility). Actual authorization enforced via `lib/auth.ts` |
